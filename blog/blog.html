Silent scopes

What is "scope" in JavaScript?
Scope is a core concept of many programming languages. It basically means "a self-sufficient block", or "an independent unit" of code. It is important when declaring variables : you have to be sure that the functions using them are in the same scope, or nothing will work! However, JavaScript puts a slight twist on this concept, by only having two kind of scopes : the global scope and the local scope. It is as simple and as complex as it sounds.

The global scope is exactly that : global. The variables declared in the global scope are accessible throughout all the code, without exceptions. They are usually placed at the top of the document, out of a function. But be careful! If you name a variable without declaring it with var, it will become global however you place it in the document!

The local scope is slightly more tricky. Every new function creates a new local scope : variables declared between the {} elements of a function are only ever accessible there. But there is a catch : any nested function inside those initial {} will also be inside that local scope AND define a new local scope of their own. And all of this is still enclosed inside the global scope.

It is just like Matryoshka doll : the global scope is the largest doll, while local scopes are all the smallest dolls inside. The catch is that there isn't necessarily only one doll inside the previous, larger doll! It can become quite confusing, which is why it is important to be very rigorous when defining the scope of your variables, to avoid unforeseen conflicts and bugs.

5 one-line CSS tricks that make your page look more designed than it really is

 Web design is an actual full-time job. But it only takes a few little tweaks to get your page to look exponentially better than it has any right to be. This is especially true when your page has lots of text. Here are five tricks I thought were amazing when I found out about them!

Change the background color
Patrick Bateman would agree : nothing is more classy than a subtle off-white coloring. And that's what your text should be displayed on. On this page, it looks like this : background-color: #f8f8f8.

Change the font's color
But you don't want that your text to be harshly black all the time! A nice gray is easier on the eye, and makes black more powerful when it is used, as you can see in my titles and links. I use color: #555 for my bodies of text.

Change the line height
Design is about balance. You need some white space to let your elements breath, or no-one will want to read your blog posts! Here, you are feasting your eyes on line-height: 1.8.

Use borders instead of underline on your links
As an extension of the previous observation about balance, look at how cramped underlined text look. Compare it to text with its bottom border showing. How distinguished! You too can make your links airier by using border-bottom: 1px solid black in the a CSS element. On my page, I even used dotted instead of solid to make it lighter still! And you can also play with the color as you wish.

Change the text selection color
Everything is in shades of gray in our design. How jarring when you select some text and it's all blue! I adressed this problem on this page with ::selection{background-color: #bababa}. More grayish goodness!

Language and signs

Punctuation has a very specific role in JavaScript, or indeed any programming language. Here are the uses for the most important punctuation signs.

Parentheses ()

Parentheses are mostly used to define parameters, arguments and conditions for functions, loops, conditional statements and methods. But they are also used to group operations together to organize their priorities.

  • Define a function's arguments :
    var myExample = function(arg){return arg * 2);}
  • Execute a function :
    myExample(2);
  • Define a loop's conditions :
    for (var i = 0; i < 10; i++){console.log(i);}
  • Define a conditional statement :
    if (i < 10){return i;}
  • Organize the priorities of arithmetic operations :
    var total = (1+2) + (2+3);
Brackets []

Brackets are all about organization of data. They define arrays and are used to access values through their index or name.

  • Define an array :
    var arr = [1,2,3];
  • Access an array's value through its index number :
    var getOne = arr[0];
  • Access an object's property value through its name :
    var obj = {number1: "one", number2: "two"};
    var getOne = obj["number1"];
Braces {}

Braces define blocks of code that are mostly independent one from another. Code between braces is usually a cohesive ensemble. Braces play a large role in defining the scope of variables : one that is defined inside braces can only be accessed there. Some examples used may seem familiar : they were already used in the parentheses section. This is because braces and parentheses very often go together : defining the condition (with ()) in which the actions will be executed (with {}). We also see our object from the bracket section again!

  • Define the code a function executes :
    var myExample = function(arg){return arg * 2);}
  • Define the code executed every time a loop repeats itself :
    for (var i = 0; i < 10; i++){console.log(i);}
  • Define the code executed if a conditional statement evaluates to true :
    if (i < 10){return i;}
  • Define an object (literal notation):
    var obj = {number1: "one", number2: "two"};

Single quotes '' and double quotes ""

In most cases, single and double quotes have exactly the same role : to define strings. The use of one or the other is down to personnal preference. However, we will see some example in which both are necessary. And also an example where they are used to access an object property's value.

  • Define a string :
    var hello = "Hello World";
    var hello2 = 'Hello World';
  • Define a string that contains single or double quotes, three posibilities :
    var bonjour = "'Bonjour' World"
    var bonjour2 = '"Bonjour" World'
    var bonjour3 = "\"Bonjour\" World"
  • Access the value of an object's property :
    var obj = {number1: "one", number2: "two"};
    var getOne = obj["number1"];

What does .map() do?
It is a simple way of applying the same change to every value stored inside an array, without deleting the old values, and without changing their order. .map() actually creates a whole new array after applying the changes specified to each and every value of the old array. In practice, it looks like this : var newArray = oldArray.map(myMapFunction(argument)). This code takes every single value indexed in my oldArray and uses myMapFunction on them to create my newArray. It works because the argument of myMapFunction will automatically be set to each of the elements of the oldArray!

What does .filter() do?
Contrary to .map(), the name is very self-explanatory : it applies a conditionnal filter to an array, and produces a new array populated with the value that passed the test (returned true). It looks like this var filteredArray = unfilteredArray.filter(myFilterFunction). Just like .map(), the argument of myFilterFunction is automatically set to the each of the values of unfilteredArray. If we have function myFilterFunction(argument) {argument = 0;}, then filteredArray will only have the values from that are 0.

What does .reduce() do?
It's not quite as obvious as .filter(), but not quite as obscured as .map() : it take the values of an array, and reduce them to one value. You can add all the value to get their sum, or you can append a series of strings to one another and get one big string. The way it works is that it takes each value of an array from left to right, executes a chosen function on them, and return the result.

JavaScript and Mental calisthenics

How does JavaScript compare to HTML and CSS?
It's harder. Waaaaay harder. But, to give a more useful answer... HTML and CSS can be compared to some kind of craftsmanship, where you physically feel your product evolving under the successive touches you progressively add to your work. It more or less behaves like you unconsciously expect it to behave. And even when it doesn't, it is usually pretty easy to locate and rectify the problem, as everything is structured in an immediately visible way. JavaScript is actual programming. You are not crafting an object, you are designing processes, things in motion, active. Which means that it is hard to actually take the measure of your progress, and to locate where your code has gone wrong. You have to adapt the way you're thinking to that flow. It can be tricky.

Explain control flow and loops using an example process from everyday life, for example 'waking up' or 'brushing your teeth'.
As I said above, JavaScript, and all other programming alnguages, are things of motion and processes, flow and loops. Let's take doing the dishes, for example. You have a starting state : a pile of dirty dishes. And you have a desired end state : a pile of clean dishes. What your code wants to do is turn the former into the latter. The "flow" of the code, or its sequential order, would go like this :
• Start task
•Go through the dishes to make sure it is dirty (this is the first loop : you look at each dish and define if it is dirty or clean)
• If it is dirty, scrub it until it is clean (the "if" and "until" words are very important, they control the flow of the code by formulating conditions to be fulfilled)
• If it is clean (which it would be after the scrubbing), put it away
• Loop back to the start and find a new dirty dish
• When all dishes are clean, stop the loop
• End task

Explain the difference between accessing data from arrays and object literals.
An array stores multiple values is a single place, in a very organized way. When you store a value in an array, it is indexed. This is to say, it is given a number that defines its place in the array. In the array var fruits = ["apple", "banana", "peach"], "apple" has the index number [0] (computers start counting at 0, not 1!) and "peach" has [2]. To access the values of an array, we would therefore logically call up their index number. If we want to access the value "banana", we would use fruits[1], calling the second value in the "fruits" array.
An object is almost the same, because it also stores multiples values in the same place. But the way it does it is by organizing every element in pairs name:value, like so : var groceryList {fruits: ["apple", "banana", "peach"], meat: "beef", drink: "Sprite"}. Every "name" ("fruits", "meat", etc) has its value ("beef", "Sprite"). It can even have an array as value (["apple", "banana", "peach"]). And you'll notice that an object literal uses curly braces {} and not brackets []. But the big difference is that objects literals have no index : we can't access the values this way. Instead, we use the "names" as keys to the value. For example, to access the value "beef, we won't write groceryList[1], but groceryList[meat]!

Explain what functions are and why they are useful.
Function are essentially a lazy man's tool. They are designed to execute a chunk of code every time they are invoked, like magic spells. You create them by giving them a name and defining their parameters (the variables it will work with), and then put the code you want executed between curly braces {}. Here is a function called repeat that will print out your argument three times every time you invoke it : function repeat(parameter){console.log(parameter + " " + parameter + " " + parameter );}
// invoking the function
repeat("bla");
//result in console
bla bla bla
Of course, there are many, many ways to use this basic capability of JavaScript, but what is important to rmember is that a function can be made every time something gets a little too repetitive during coding. A golden rule is to never repeat yourself!

Be responsive to the wireframe

As we've seen two days ago, a responsive website adapts to whatever display or device you're using to browse it. It is tremendously important, because the days when the Web could only be accessed through a laptop computer with standardized 4/3 or 16/9 monitors are gone. It is therefore crucial to make it so your Web content can be displayed just as well on any screen size, on any device, and through any browser. It doesn't mean that it must look exactly the same, but it does mean it should be as legible and as good looking.

With this in mind, "mobile first" design is getting more and more popular. It is hard to shrink an interface created for a large screen. And it is a hassle to maintain a mobile AND a desktop site. So, the simplest solution is to start with the mobile site, and design so that it can expand gracefully to any screen size.

This can be a little tricky. Designing responsive website is a whole lot more complexe than designing for only one screen size. That's where frameworks come in handy : some can offer to do all the structural heavy lifting for you. They give you access to the blueprint of a website, with the corresponding code to use. It can save you many hours of frustrating experimentations with complex CSS properties (as I well know...) However, there is a price, which is creativity : you have to use the framework's structure. It gets very, very complex when you try to make it do something for which it is not designed.

Wireframing is basically boxifying, but at a more conceptual level. It happens before starting to think of the technical solutions : it is akin to the plans an architect would draw up. We use it to define the features of our websites, and their spatial organization on the page. Wireframing is the third phase of any kind of web design, and answers the question "How will it look?". It comes after the information gathering and analysis ("Why do we build it?), the technical planning ("How will we build it?").

Responsive griddiness

What is grid based design? You remember how the Internet is all boxes?
Well, grid based design was invented to neatly arrange these boxes in exactly the way we want them. It is a CSS design philosphy, based on the the box model. Concretely, this means dividing our web page in columns (usually 12) and rows (usually occupying 100% of the width of the page). With these two variables, we can do almost any kind of layout.

Whats all the hype about responsive webpages?
In today's world, only half the page views you get are from computer (desktop or laptop), with traditionnal 16/9, 1080p displays. The other half is made up of all the tablets, phones and more obscure mobile devices used to browse the Web. Responsive design seeks to provide an experience of the same quality for any device, and that means gracefully and dynamically adapting to any screen size, and any aspect ratio.

What is semantic structure?
Semantic literally means "meaning". Anything that is "semantic" seeks to be understandable by humans in an explicit sense, and not through knowledge of esoteric rules. In the case of semantic structure, it refers to the naming of specific elements in the HTML structure : the "headlines" <h1>, <h2> and <h3>; of decreasing importance or the "paragraph" <p>, for example. They are different in that they are not generic division of space, but have a meaning embedded into their code, and should be used accordingly.

Boxifying everything

What is meant by boxifying design?
Boxifying is the act of taking a design and turning it into a series of boxes through the tools of HTML and CSS. Everything on the Internet can be reduced to a box, whether large or small. In fact, most boxes on the internet are inside larger boxes, and contain smaller boxes : every web page is a matryoshka doll!

What is the box model?
The box model is a shorthand used to describe the design elements that define every HTLM box. From larger to smaller, it goes : margins > borders > padding > content. Each one of these elements can be modified through CSS to achieve the desired design.

Mindfulness, attention, emotional intelligence (Search Inside Yourself, Chapter 1)

I have been aware of the powers of mindfulness for a while, now. Ten years ago, I developped chronic tinnitus from too many loud concerts. It has worsened several times since that day, for reasons both in and out of my control. And there have been very difficult and stressful days from that little ringing. But one thing I have experienced first-hand is how much better practicing mindfulness made everything. I saw behavioural therapist for my tinnitus issues several times, and every time their treatment was based on mindfulness : just "being", as Chade-Meng Tan puts it. It was hard at first, as the ringing got louder and louder, clamoring for my full attention, and therefore triggering a panic response. But, keeping at it, not even very often, or for very long... I noticed that it was getting better. I accepted the ringing with more neutrality. It was one of the many stimuli coming to my attention, just like the breath in my chest or the ground against my feet. And eventually, in my day-to-day life, the ringing got exactly the same amount of attention that those other two stimuli got. Which is to say, almost none.

This very deep personal experience makes me extremely enthusiastic about what Search Inside Yourself is all about. The way the program is laid out feels very attractive, and not necessarily only in a professional context. I'm like Chade-Meng in that what is the most important for me is happiness, for me and others. And I absolutely understand how practicing mindfulness, honing your attention and developing emotional intelligence could be a major help in achieving happiness. Even if you end up unemployed!

I thoroughly enjoyed the two minutes of mindfulness following the reading of this first chapters. It complimented nicely the 10 minutes of guided meditation I did this morning with the Headspace app! I'll try to form this habit from now on, practicing mindfulness twice a day, after significant milestones in my day.

Time and habits management

which timebox and habits techniques did you try?
I've never used any timebox techniques consciously, but I feel that the pomodoro technique is something I unconsciously applied to a lot of work I did during my studies, especially when writing especially long papers. Pausing for 10 minutes avery half hour sounds exaactly like my workflow whe doing that kind of tasks. I don't know if it will apply to dev work though! I also tried to establish a meditation habit several times, but it always tapered off at some point. I'll keep trying until I man age to get it, though!

What did you find interesting?
The most interesting element I encountered was the "Trigger" concept for creating habit. I think it was the one thing I missed to actually make it work, and the one thing that made things work unconsciously!

What worked for you?
A while ago, I managed to create a habit of running with the trigger "Go running immediately after you wake up" a while ago, without being aware of the "trigger" aspect. Very interesting to know that it was this element that made it so effective.

Which will you experiment further with?
I will definitely try to get more into the habit of timeboxing tasks with the pomodoro technique, and using the breaks to get into the habit of meditating. I actually have downloaded the Headspace app! Will use it for the first time after I push this blog entry :p
