The time I got blocked on the Profile lookup challenge of Freecodecamp basic JS course
This course had realy unexpected difficulty spikes, namely with the Record collection and the Profile lookup challenges. Both demanded "complex" articulations of the "simple" principles I had been learning about. It is very apparent in the Profile lookup challenge, which called for the use of a for loop inside a for loop. For some reason, I had a very hard time to wrap my head around exactly how a loop inside a loop works. I kept trying things, and nothing worked as I expected it to work. This is the basis of all frustration : you expect something from the world, and it gives you something else (or nothing!). This is why devotes Mac, Linux or Windows users forced to use another platform are so vocally displeased : they keep expecting the system to behave a certain way, and keep getting thwarted. It was through an actual conversation with a dev friend that I was able to grasp how, with nested loops, the inner loop must finish before the outer loop increments. I bounced all my ideas off her, and she explained to me why they were wrong, guiding me to the right way to think about the problem. Being able to have a reliable human source of information, not just written documentation, was incredibly valuable. I felt comfortable in my ignorance and confident in my ability to learn while we were conversing. It helped me tremendously.

What does .map() do?
It is a simple way of applying the same change to every value stored inside an array, without deleting the old values, and without changing their order. .map() actually creates a whole new array after applying the changes specified to each and every value of the old array. In practice, it looks like this : var newArray = oldArray.map(myMapFunction(argument)). This code takes every single value indexed in my oldArray and uses myMapFunction on them to create my newArray. It works because the argument of myMapFunction will automatically be set to each of the elements of the oldArray!

What does .filter() do?
Contrary to .map(), the name is very self-explanatory : it applies a conditionnal filter to an array, and produces a new array populated with the value that passed the test (returned true). It looks like this var filteredArray = unfilteredArray.filter(myFilterFunction). Just like .map(), the argument of myFilterFunction is automatically set to the each of the values of unfilteredArray. If we have function myFilterFunction(argument) {argument = 0;}, then filteredArray will only have the values from that are 0.

What does .reduce() do?
It's not quite as obvious as .filter(), but not quite as obscured as .map() : it take the values of an array, and reduce them to one value. You can add all the value to get their sum, or you can append a series of strings to one another and get one big string. The way it works is that it takes each value of an array from left to right, executes a chosen function on them, and return the result.

JavaScript and Mental calisthenics

How does JavaScript compare to HTML and CSS?
It's harder. Waaaaay harder. But, to give a more useful answer... HTML and CSS can be compared to some kind of craftsmanship, where you physically feel your product evolving under the successive touches you progressively add to your work. It more or less behaves like you unconsciously expect it to behave. And even when it doesn't, it is usually pretty easy to locate and rectify the problem, as everything is structured in an immediately visible way. JavaScript is actual programming. You are not crafting an object, you are designing processes, things in motion, active. Which means that it is hard to actually take the measure of your progress, and to locate where your code has gone wrong. You have to adapt the way you're thinking to that flow. It can be tricky.

Explain control flow and loops using an example process from everyday life, for example 'waking up' or 'brushing your teeth'.
As I said above, JavaScript, and all other programming alnguages, are things of motion and processes, flow and loops. Let's take doing the dishes, for example. You have a starting state : a pile of dirty dishes. And you have a desired end state : a pile of clean dishes. What your code wants to do is turn the former into the latter. The "flow" of the code, or its sequential order, would go like this :
• Start task
•Go through the dishes to make sure it is dirty (this is the first loop : you look at each dish and define if it is dirty or clean)
• If it is dirty, scrub it until it is clean (the "if" and "until" words are very important, they control the flow of the code by formulating conditions to be fulfilled)
• If it is clean (which it would be after the scrubbing), put it away
• Loop back to the start and find a new dirty dish
• When all dishes are clean, stop the loop
• End task

Explain the difference between accessing data from arrays and object literals.
An array stores multiple values is a single place, in a very organized way. When you store a value in an array, it is indexed. This is to say, it is given a number that defines its place in the array. In the array var fruits = ["apple", "banana", "peach"], "apple" has the index number [0] (computers start counting at 0, not 1!) and "peach" has [2]. To access the values of an array, we would therefore logically call up their index number. If we want to access the value "banana", we would use fruits[1], calling the second value in the "fruits" array.
An object is almost the same, because it also stores multiples values in the same place. But the way it does it is by organizing every element in pairs name:value, like so : var groceryList {fruits: ["apple", "banana", "peach"], meat: "beef", drink: "Sprite"}. Every "name" ("fruits", "meat", etc) has its value ("beef", "Sprite"). It can even have an array as value (["apple", "banana", "peach"]). And you'll notice that an object literal uses curly braces {} and not brackets []. But the big difference is that objects literals have no index : we can't access the values this way. Instead, we use the "names" as keys to the value. For example, to access the value "beef, we won't write groceryList[1], but groceryList[meat]!

Explain what functions are and why they are useful.
Function are essentially a lazy man's tool. They are designed to execute a chunk of code every time they are invoked, like magic spells. You create them by giving them a name and defining their parameters (the variables it will work with), and then put the code you want executed between curly braces {}. Here is a function called repeat that will print out your argument three times every time you invoke it : function repeat(parameter){console.log(parameter + " " + parameter + " " + parameter );}

Be responsive to the wireframe

As we've seen two days ago, a responsive website adapts to whatever display or device you're using to browse it. It is tremendously important, because the days when the Web could only be accessed through a laptop computer with standardized 4/3 or 16/9 monitors are gone. It is therefore crucial to make it so your Web content can be displayed just as well on any screen size, on any device, and through any browser. It doesn't mean that it must look exactly the same, but it does mean it should be as legible and as good looking.

With this in mind, "mobile first" design is getting more and more popular. It is hard to shrink an interface created for a large screen. And it is a hassle to maintain a mobile AND a desktop site. So, the simplest solution is to start with the mobile site, and design so that it can expand gracefully to any screen size.

This can be a little tricky. Designing responsive website is a whole lot more complexe than designing for only one screen size. That's where frameworks come in handy : some can offer to do all the structural heavy lifting for you. They give you access to the blueprint of a website, with the corresponding code to use. It can save you many hours of frustrating experimentations with complex CSS properties (as I well know...) However, there is a price, which is creativity : you have to use the framework's structure. It gets very, very complex when you try to make it do something for which it is not designed.

Wireframing is basically boxifying, but at a more conceptual level. It happens before starting to think of the technical solutions : it is akin to the plans an architect would draw up. We use it to define the features of our websites, and their spatial organization on the page. Wireframing is the third phase of any kind of web design, and answers the question "How will it look?". It comes after the information gathering and analysis ("Why do we build it?), the technical planning ("How will we build it?").

Responsive griddiness

What is grid based design? You remember how the Internet is all boxes?
Well, grid based design was invented to neatly arrange these boxes in exactly the way we want them. It is a CSS design philosphy, based on the the box model. Concretely, this means dividing our web page in columns (usually 12) and rows (usually occupying 100% of the width of the page). With these two variables, we can do almost any kind of layout.

Whats all the hype about responsive webpages?
In today's world, only half the page views you get are from computer (desktop or laptop), with traditionnal 16/9, 1080p displays. The other half is made up of all the tablets, phones and more obscure mobile devices used to browse the Web. Responsive design seeks to provide an experience of the same quality for any device, and that means gracefully and dynamically adapting to any screen size, and any aspect ratio.

What is semantic structure?
Semantic literally means "meaning". Anything that is "semantic" seeks to be understandable by humans in an explicit sense, and not through knowledge of esoteric rules. In the case of semantic structure, it refers to the naming of specific elements in the HTML structure : the "headlines" <h1>, <h2> and <h3>; of decreasing importance or the "paragraph" <p>, for example. They are different in that they are not generic division of space, but have a meaning embedded into their code, and should be used accordingly.

Boxifying everything

What is meant by boxifying design?
Boxifying is the act of taking a design and turning it into a series of boxes through the tools of HTML and CSS. Everything on the Internet can be reduced to a box, whether large or small. In fact, most boxes on the internet are inside larger boxes, and contain smaller boxes : every web page is a matryoshka doll!

What is the box model?
The box model is a shorthand used to describe the design elements that define every HTLM box. From larger to smaller, it goes : margins > borders > padding > content. Each one of these elements can be modified through CSS to achieve the desired design.

And eventually, in my day-to-day life, the ringing got exactly the same amount of attention that those other two stimuli got. Which is to say, almost none.

This very deep personal experience makes me extremely enthusiastic about what Search Inside Yourself is all about.
